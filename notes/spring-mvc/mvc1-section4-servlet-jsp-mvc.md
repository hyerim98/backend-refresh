# MVC 아키텍처 발전 과정 (Servlet → JSP → MVC Pattern)

## 1. 순수 서블릿 방식

### 특징
- HttpServlet을 사용하여 모든 로직 처리
- HTML을 자바 코드 안에서 직접 작성

### 문제점
- 비즈니스 로직 + 뷰 렌더링 코드 혼재
- 가독성 매우 낮음
- 유지보수 어려움
- 역할 분리가 안 됨

### 한계
→ 관심사 분리가 필요함

---

## 2. JSP 도입

### 목적
- 뷰(View)를 JSP로 분리
- HTML 작성 편의성 향상

### 구조
- 서블릿: 비즈니스 로직 처리
- JSP: 화면 렌더링

### 개선점
- HTML 코드 분리
- 화면 작성이 쉬워짐

### 문제점
- 여전히 JSP에 자바 코드(scriptlet) 포함
- 컨트롤러 역할이 명확하지 않음
- 책임이 완전히 분리되지 않음

---

## 3. MVC 패턴 등장

### 등장 배경
- 역할과 책임을 명확히 분리하기 위해

---

## MVC 구성 요소

### 1) Controller
- 요청을 받음
- 비즈니스 로직 호출
- 모델에 데이터 저장
- 뷰 이름 반환

### 2) Model
- View에 전달할 데이터 저장
- request.setAttribute() 사용

### 3) View (JSP)
- 모델 데이터를 화면에 렌더링
- 비즈니스 로직 없음

---

## MVC 패턴의 장점

- 관심사 분리 (Separation of Concerns)
- 유지보수 용이
- 테스트 용이
- 확장성 증가

---

## 흐름 정리

클라이언트 요청
→ Controller
→ 비즈니스 로직
→ Model 데이터 저장
→ View(JSP) 렌더링
→ 응답 반환

---

## 핵심 키워드

- 관심사 분리
- 역할 분리
- 유지보수성 향상
- 확장 가능 구조

---

## 왜 MVC가 중요한가?

이 구조가 이후:

- 프론트 컨트롤러 패턴
- 스프링 MVC
- REST API 설계

로 확장됨

MVC는 모든 웹 프레임워크의 기본 구조이다.
