# 섹션4 객체지향 원리 적용 – AppConfig와 DI

## 섹션3의 문제점
섹션3에서는 App이 구현체를 직접 생성했다.
이로 인해 다음과 같은 문제가 있었다.

- DIP 위반: 인터페이스가 구현체에 의존
- OCP 위반: 구현 변경 시 App 코드 수정 필요
- 객체 생성과 사용 책임이 섞여 있음

## 섹션4의 핵심 변화
섹션4에서는 객체 생성 책임을 AppConfig로 분리했다.

- App은 이제 역할(인터페이스)만 의존
- 어떤 구현체를 사용할지는 AppConfig에서 결정
- 애플리케이션이 조립된다는 개념 이해

## AppConfig의 역할
- 객체 생성 책임을 한 곳에 모음
- 구현 변경 시 App 코드 수정 불필요
- 역할과 구현의 명확한 분리

## 객체지향 원리 관점에서의 의미
- DIP: 고수준 모듈이 구현체에 의존하지 않음
- OCP: 기존 코드 수정 없이 확장 가능

## 개인적인 깨달음
new를 어디서 하느냐가 설계의 핵심이라는 걸 처음 체감했다.
객체지향 원리는 이론이 아니라 코드 구조로 드러난다는 걸 알게 됐다.
