# 섹션7. 컴포넌트 스캔과 자동 의존관계 주입

## 왜 컴포넌트 스캔이 필요한가?
- AppConfig에 @Bean을 하나하나 등록하는 방식의 한계
- 클래스 수가 많아질수록 설정 코드가 비대해짐
- 설정 실수 가능성 증가

👉 해결: 스프링이 직접 빈을 찾아서 등록하게 하자

## 컴포넌트 스캔이란?
- @Component가 붙은 클래스를 자동으로 스프링 빈으로 등록
- @ComponentScan을 통해 지정한 패키지부터 탐색
- **기본적으로 @SpringBootApplication 내부에 포함되어 있음**

## @Component 계열 애노테이션
- @Component: 일반 컴포넌트
- @Service: 비즈니스 로직
- @Repository: 데이터 접근 계층
- @Controller / @RestController: 웹 계층

※ 기능은 동일하지만 역할을 명확히 드러냄

## 자동 의존관계 주입 (@Autowired)
- 생성자에 @Autowired를 붙이면 스프링이 자동으로 빈을 주입
- 생성자 1개일 경우 @Autowired 생략 가능
- 타입 기준으로 먼저 탐색 → 이름으로 보조 매칭

## 컴포넌트 스캔 + DI의 효과
- 설정 코드(AppConfig)가 거의 사라짐
- 구현체 변경 시 코드 수정 없이 교체 가능
- DIP, OCP를 스프링이 대신 지켜줌

## 주의할 점
- 컴포넌트 스캔 범위가 넓으면 불필요한 빈 등록 가능
- 같은 타입의 빈이 여러 개면 충돌 발생 가능
- 명시적 설정(@Bean)과 자동 등록 혼용 시 우선순위 이해 필요

## 정리
- 컴포넌트 스캔은 설정 자동화
- @Autowired는 의존성 주입 자동화
- 스프링이 객체 생성과 연결을 완전히 책임짐
